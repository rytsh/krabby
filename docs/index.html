// what it does

Built for agents that read code all day.

Every capability below is served straight to your LLM or agent over a single streamable-HTTP MCP endpoint.

01

Queries are fast

Graph query tools are answered in-process by a native Go engine that reads graph.json directly and hot-reloads on rebuild — no per-graph subprocess is spawned.

query_graph · get_node · shortest_path
02

Builds are cheap

Code extraction is AST-based — no LLM key needed. The graphify CLI is only invoked to build graphs incrementally.

graphify update
03

Docs & RAG

With an LLM configured, krabby generates per-file Markdown docs plus a repo overview — browsable in the UI. The prompt is configurable.

optional · search_docs
04

Web sources

Custom web URL collections and Confluence spaces are converted to Markdown and indexed beside repo docs. Search one collection like web:wine or everything at once.

list_sources · confluence
05

Semantic code search

Source is chunked at graphify symbol boundaries, embedded with a dedicated code model such as Codestral Embed, and returned as ranked path/line snippets.

optional · code/search
06

Graph-backed code search

Find real code through the repository graph: symbols, definitions, usages and dependencies are resolved to the exact path and line.

search_code · query_graph
// how it fits together

One Go binary between agents and repos.

krabby · pipeline
+---------------------------------------+
| LLM/Agent   --MCP--> krabby (Go)       |
| CI/webhook   --HTTP-> krabby (Go)       |
|                                       |
|  |- MCP tools (manage + query)          |
|  |- REST API + provider-neutral hook    |
|  |- Registry (bw/BadgerDB)              |
|  |- Native graph query engine (Go)    |
|  `- Scheduler (poll interval)           |
+---------------------------------------+
  • Registry in embedded BadgerDB (bw) tracks every repo and credential.
  • Plain files on disk under ~/.krabby — clones, graph.json, reports and vectors are all readable by other tools.
  • Provider-neutral webhook at /webhook/git accepts common server formats with bearer / shared-token auth.
// get running

From clone to first query in a minute.

Requires git, ssh, and the graphify CLI (uv tool install graphifyy).

shell — release & run
# download the latest release
https://github.com/rytsh/krabby/releases/latest

# configure and start the server
 cp krabby.example.yaml krabby.yaml   # edit repos, keys
 ./bin/krabby

# track a repo, then list what you have
curl -X POST localhost:8080/api/v1/repos \
  -d '{"url":"git@git.example.com:team/service.git"}'
curl localhost:8080/api/v1/repos
LLM — MCP setup prompt
Set up Krabby as an MCP server for this repository.

Download the latest Krabby release, start the server, and connect
the MCP client to:
http://localhost:8080/mcp
Use this header:
X-Krabby-Tool-Profile: full

After Krabby is running, configure its runtime model settings either
through the Krabby web UI at http://localhost:8080 or through MCP tools:
- LLM provider/model and API key for generated code documentation
- General embedder for document/RAG search
- Code embedder for code indexing and semantic code search
Run test_llm, test_embedder, and test_code_embedder before adding or
refreshing repositories so indexing and docs generation are verified.

Verify that the MCP connection works, then use Krabby for this
repository. For code questions, use graph-backed search first:
find symbols, definitions, usages and dependencies through the
repository graph, and always report exact file paths and lines.
